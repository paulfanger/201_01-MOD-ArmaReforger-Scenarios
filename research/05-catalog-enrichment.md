# Research 05: Catalog Enrichment — Faction & Loadout GUIDs

**Date:** 2026-05-20
**Status:** Complete

## Summary

Searched all four cloned OSS repos in `/tmp/reforger-research/` for US/USSR faction prefabs, loadout configs, character prefabs, and AI group prefabs. Ran comprehensive gap analysis against the existing catalog (1326 entries). Implemented semantic alias layer to resolve `ENF_Faction_US` / `ENF_Faction_USSR` refs generated by narrative-designer.

---

## Key Finding: Catalog Was Already Complete

The bootstrap (run previously) had already captured all US/USSR/USMC faction data found in the repos. Comprehensive gap analysis showed:

- 22 US/USSR character prefabs: **all in catalog**
- 54 group prefabs across all factions: **all in catalog**
- 10 US/USSR loadout confs: **all in catalog**

The actual problem was a **semantic layer gap**: the `narrative-designer` generates `ENF_Faction_US` and `ENF_Faction_USSR` as `asset_id_ref` values, but no catalog entry existed with those names. The `asset-curator` correctly flagged them as `missing_assets` with `halt_required = false`.

## Changes Made

### 1. `catalog/ALIASES.json` (new file)
Semantic alias registry mapping `narrative-designer` asset_id_ref values to catalog GUIDs:
- `ENF_Faction_US` → `ADFDBDA163950168`
- `ENF_Faction_USSR` → `15B582F8FA0B0940`
- `ENF_Faction_FIA` → `CF9447B87AB774DB`

### 2. `catalog/INDEX.json` (enriched)
Added `semantic_aliases` and `aliases_file` fields, preserved through future bootstrap runs.

### 3. Catalog entries enriched (50 total)
- 3 faction conf entries: added `faction_key`, `faction_side`, `class_hint`, `aliases`
- 47 character/group/loadout entries: added `faction_key` field

### 4. `backend/catalog/resolver.py` (updated)
Added:
- `resolve_alias(alias_name)` — resolves `ENF_Faction_US` → full catalog entry
- `list_aliases()` — returns all known alias mappings
- Alias loading from `ALIASES.json` on init (with INDEX fallback)

### 5. `backend/catalog/bootstrap.py` (updated)
Bootstrap now **preserves enrichment fields** when re-run:
- Reads existing JSON before writing, merges enrichment keys back
- Preserves `semantic_aliases` and `aliases_file` in INDEX

### 6. `backend/catalog/enrich_factions.py` (new script)
Re-runnable enrichment script. Run after bootstrap if enrichment is lost:
```bash
python3 backend/catalog/enrich_factions.py
```

## Verification

```
python3 -c "
from backend.catalog.resolver import CatalogResolver
r = CatalogResolver()
us = r.resolve_alias('ENF_Faction_US')
ussr = r.resolve_alias('ENF_Faction_USSR')
print(us['guid'], us['faction_key'])   # ADFDBDA163950168 US
print(ussr['guid'], ussr['faction_key'])  # 15B582F8FA0B0940 USSR
"
```

Both resolve correctly. The `test-mission-pipeline-check` asset-manifest `missing_assets` for `ENF_Faction_US` / `ENF_Faction_USSR` can now be resolved by updating `asset-curator` to call `resolve_alias()` before falling through to display_name search.

- `faction_key "US"` and `"USSR"` are the Reforger engine string identifiers for each faction — they match the `m_sPlayerFactionKey`/`m_sEnemyFactionKey` values in FactionManager prefabs.
- USMC (RHS mod) faction key is `"RHS_US_USMC"` — not vanilla, not needed for MVP.
- The vanilla faction hierarchy: `US_Campaign.conf` inherits from `Configs/Factions/BLUFOR.conf` (GUID `E7B7EA4823DD81F0`). Both are in catalog.
